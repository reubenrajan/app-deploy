Discipline A — AgPT
Agentic Performance Testing
Testing the performance of AI agents themselves — measuring how fast, how reliably, and how cost-effectively autonomous agents complete multi-step tasks. The agent is the system under test. You validate latency across reasoning chains, tool-call overhead, token throughput, memory retrieval speed, and end-to-end task completion time.
AI is the subject under scrutiny
Discipline B — ADPT
Agent-Driven Performance Testing
Using AI agents as the testing instrument — deploying autonomous agents to generate, execute, and analyze performance tests against traditional software systems. The agent is the tester. It writes load scripts, spawns virtual users, interprets results, and self-heals test scenarios based on observed system behaviour.
AI is the tool accelerating tests

Key Differences at a Glance
Dimension Agentic Performance Testing (AgPT) Agent-Driven Performance Testing (ADPT)
What is being testedThe AI agent / workflow itself — agent is the system under testA software system (API, DB, microservice) — agent is the tester
Role of AI in the workAI is the subject under scrutiny — measured, profiled, benchmarkedAI is the executor and co-pilot — generates, runs, and interprets tests
Primary metric vocabularyTTFT, TPOT, tokens/request, cost/completion, task success rate, eval scoreTPS, VU count, p95/p99 latency, error rate, script coverage
Key failure modesHallucination under load, loop divergence, context truncation, rate limit cascadesTimeout storms, connection pool exhaustion, DB lock contention, bad scripts in prod
Core toolchainLangSmith, Langfuse, Arize, DeepEval, RAGAS, promptfoo, LiteLLMk6, Gatling, Locust + LangChain/AutoGen for orchestration + LLM APIs
Determinism of resultsProbabilistic — variance across identical runs is expected; statistics are mandatoryLargely deterministic — scripts are stable; variance is system noise, not tool noise
Hardest skill to acquireReasoning chain profiling + cross-model benchmarking + eval design under non-determinismAdaptive load shaping + safe autonomous execution + trust calibration
Risk if done poorlyInvisible cost overruns, unpredictable agent latency, quality regressions reaching usersAgents generating misleading scripts or — worst case — injecting load into production
Career adjacencySRE, ML engineering, AI product reliabilitySDET, QE automation, AI-augmented DevOps
Practice maturityEmerging specialist discipline — no established playbook, high demand, low supplyEvolution of an existing role — natural progression for performance engineers upskilling into AI
When to prioritise AgPT skills
Your organisation is shipping AI-powered products — agents, copilots, or RAG workflows — where performance, reliability, and cost are customer-facing concerns. You need to define SLOs for AI behaviour, not just infrastructure uptime. The people who can do this are rare and valuable; the toolchain is still forming and the practitioner who can also build it will have outsized influence.
When to prioritise ADPT skills
Your organisation wants to dramatically accelerate how performance tests are created and maintained — reducing the manual scripting burden, expanding coverage without growing headcount, and getting intelligent analysis without waiting for an expert to read charts. This is the lower barrier path: existing performance engineers can adopt ADPT incrementally without rebuilding their entire practice.
